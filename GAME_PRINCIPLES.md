# GAME PRINCIPLES — 미니게임 제작 원칙

이 문서는 `game-portal`에 간단한 게임을 계속 추가하기 위한 기준이다. 목표는 게임 수를 늘리는 것이지만, 더 중요한 것은 모든 게임이 같은 포털의 일부처럼 느껴지고, 짧은 시간 안에 바로 재미를 주는 것이다.

## 1. 포털 공통 원칙

### 1.1 한 제품처럼 보여야 한다

모든 게임은 독립 작품이 아니라 하나의 미니게임 포털 안에 들어온다. 그래서 아래 요소는 게임마다 새로 만들지 않고 공통 레이어를 따른다.

- 공통 헤더와 푸터
- 공통 게임 셸과 배경 톤
- 공통 사운드/음소거 위치
- 공통 기록 저장과 기록실 노출
- 공통 공유 플로우
- 공통 피드백 진입점
- 공통 후원 진입점
- 공통 아이콘/오브젝트 키트

게임마다 개성이 있어도 된다. 하지만 폰트, 버튼, 아이콘, 결과 화면, 기록 공유 방식이 제각각이면 포털이 아니라 게임 파일 모음처럼 보인다.

### 1.2 게임별 개성은 "내용"에서 나온다

일관성은 껍데기를 맞추는 일이고, 개성은 플레이와 세계관에서 나온다.

- `물병 정렬`은 차분한 퍼즐감
- `라면집 사장님`은 밀려드는 손님과 운영 압박
- `Cube Snake`는 큐브 표면을 넘나드는 공간감

새 게임도 "어떤 행동이 재미있는가", "어떤 상황을 상상하게 하는가", "왜 한 판 더 하고 싶은가"가 먼저 있어야 한다. 색만 바꾼 복제 게임은 추가하지 않는다.

아이디어가 바로 떠오르지 않으면 기존에 검증된 유명 게임을 창의적으로 비튼다. 예를 들어 뱀 게임을 3D 큐브 표면으로 옮기거나, 블록 퍼즐에 방치 가능한 성장 구조를 붙이는 식이다. 단순 복제는 안 되고, 입력 방식, 공간 구조, 목표, 세계관, 기록 단위 중 적어도 하나는 분명히 달라야 한다.

## 2. 공통 UX 구성

### 2.1 메인 화면 게임 카드

메인 화면의 각 게임 카드는 같은 정보 구조를 가진다.

- 게임 제목
- 한 줄 태그라인
- 대표 기록 단위
- 즉시 플레이 버튼
- 게임의 핵심 행동을 암시하는 시각 요소

카드는 설명서가 아니라 초대장이다. 사용자가 1초 안에 "이건 이런 게임이구나"를 알아야 한다.

### 2.2 기록실

기록실은 모든 게임의 성과가 모이는 공간이다.

- 게임별 최고 기록
- 내 기록
- 친구 또는 전체 기록
- 공유 가능한 기록 카드
- 게임으로 돌아가는 진입점

기록 단위는 게임마다 다를 수 있다. 점수, 레벨, 층, 타일 수처럼 게임성에 맞는 단위를 쓰되, `games.json`에 명확히 등록한다.

### 2.3 게임 종료 화면

게임 종료 화면은 모든 게임에서 가장 중요한 공통 루프다.

반드시 포함한다.

- 이번 판 결과
- 최고 기록 갱신 여부
- 다시 하기
- 기록 공유
- 기록실 보기
- 피드백 보내기
- 후원 진입점

게임 종료는 실패 안내가 아니라 다음 행동을 만드는 순간이다. "아깝다", "한 번 더 하면 되겠다", "공유하고 싶다" 중 하나가 떠야 한다.

### 2.4 피드백

피드백은 게임별로 흩어지면 안 된다. 공통 UI에서 게임 id, 기록, 기기 정보, 현재 URL 정도를 함께 보낼 수 있어야 한다.

피드백 진입점은 숨기지 않는다. 다만 플레이 중 집중을 방해하지 않게 결과 화면, 메뉴, 푸터에 둔다.

### 2.5 후원

후원 기능은 게임별 코드에 박지 않는다. 포털 공통 레이어가 주입한다.

- 결과 화면에 자연스럽게 노출
- 푸터 또는 메뉴에서도 접근 가능
- 토스, 카카오뱅크 등 실제 수단은 서버 설정으로 관리
- 계좌, 송금 링크, 시크릿은 저장소에 커밋하지 않는다

후원 문구는 구걸처럼 보이면 안 된다. "재미있었다면 다음 게임 제작을 도와주세요"처럼 제작 지속성과 연결한다.

## 3. 조작 원칙

### 3.1 한 손으로 가능해야 한다

기본 플레이는 모바일 한 손 조작을 기준으로 한다.

- 엄지 하나로 핵심 조작 가능
- 화면 하단부에서 주요 입력 가능
- 동시에 두 지점을 누르는 조작 금지
- 긴 드래그, 정밀 조준, 작은 버튼 남발 금지
- 세로 화면을 기본으로 고려

키보드와 데스크톱 조작은 추가로 지원할 수 있지만, 모바일 한 손 플레이가 먼저다.

### 3.2 설명 없이 시작되어야 한다

첫 판은 튜토리얼을 읽지 않아도 굴러가야 한다.

- 첫 입력 전 1개 행동만 요구
- 첫 실패는 관대하게
- 규칙은 플레이 중 자연스럽게 드러나게
- 텍스트 설명보다 배치, 움직임, 반응으로 이해시키기

"플레이 방법" 문구가 길어지면 게임 설계가 복잡하다는 신호다.

### 3.3 실력으로 플레이한다는 느낌

운이 있어도 되지만, 결과가 운만으로 보이면 안 된다. 사용자는 "내가 방금 더 잘할 수 있었다"는 감각을 받아야 한다.

좋은 실패는 이유가 보인다.

- 너무 늦게 눌렀다
- 위치를 잘못 골랐다
- 다음 수를 못 봤다
- 욕심을 냈다

나쁜 실패는 억울하다.

- 판정이 이상하다
- 갑자기 속도가 뛴다
- 화면 밖에서 당한다
- 입력했는데 반응하지 않는다
- 규칙을 몰라서 진다

### 3.4 시간제한이 없어야 한다

기본 게임 구조에 시간제한을 두지 않는다. 사용자가 게임을 켜놓고 다른 일을 하다가 돌아와도 이어서 할 수 있어야 하고, 오래 생각해도 손해를 보지 않아야 한다.

- 제한 시간으로 실패시키지 않는다.
- 방치했다고 판이 망가지지 않는다.
- 생각하는 동안 적, 게이지, 타이머가 압박하지 않는다.
- 앱을 잠깐 나갔다 돌아와도 상태가 보존된다.
- 긴 고민이 가능한 판 구조를 만든다.

긴장감은 시간 압박이 아니라 선택의 결과, 누적된 위험, 점점 어려워지는 배치에서 만든다. 이 포털의 게임은 사용자를 쫓아내지 않고 머무르게 해야 한다.

## 4. 재미의 필수 요소

### 4.1 타격감과 손맛

모든 게임은 입력에 대한 즉각적인 보상이 있어야 한다.

- 누르는 순간 반응
- 성공 시 짧고 선명한 애니메이션
- 실패 시 납득 가능한 피드백
- 콤보, 연속 성공, 아슬아슬함을 강조
- 점수 변화가 눈에 들어오게
- 사운드와 진동은 선택적으로, 하지만 있으면 반응과 정확히 맞게

타격감은 큰 이펙트가 아니라 입력과 결과 사이의 짧은 거리에서 나온다.

#### 눈이 즐거운 "주스(juice)" — 반드시 따로 고민할 것

새 게임을 설계할 때 "이 게임에서 눈 돌아가게 만들 그래픽 요소가 뭐가 있을까?"를 **별도의 질문으로 한 번 고민**하고, 게임성에 맞는 것을 **최소 3가지 이상** 넣는다. 기능이 다 돌아간 뒤에 붙이는 장식이 아니라 설계 단계의 필수 항목이다. 단, 판독성(다음 수 읽기)을 해치는 순간 뺀다.

자주 쓰는 재료:

- **이동 잔상/트레일** — 빠르게 움직이는 오브젝트 뒤에 페이드 꼬리
- **파괴 파편 + 플래시** — 부서지는 순간 조각이 튀고 한 프레임 밝게 번쩍
- **화면 흔들림** — 큰 임팩트에 2~6px 셰이크. 작게, 짧게, 콤보일수록 크게
- **콤보 연출** — 연쇄 성공 시 커지는 숫자 팝(×5, ×8…)
- **등장 바운스** — 새 오브젝트가 살짝 오버슈트하며 안착(easeOutBack)
- **임팩트 링/파동** — 충돌·수집 지점에서 퍼지는 링
- **플로팅 텍스트** — +1, ×콤보 같은 짧은 보상 숫자가 떠올랐다 사라짐
- **스쿼시&스트레치** — 부딪힌 방향으로 찌그러졌다 복원
- **신기록 전용 연출** — 평소 클리어보다 한 단계 화려하게 (여기만 과해도 됨)

### 4.2 재미있는 디자인 요소

게임 화면은 기능만 있으면 안 된다. 한두 개의 기억나는 시각 장치가 있어야 한다.

- 게임의 핵심 오브젝트
- 진행에 따라 바뀌는 상태
- 성공을 축하하는 작은 변화
- 실패 직전의 긴장감
- 신기록 순간의 특별한 연출

단, 장식이 판독성을 해치면 제거한다. 재미있는 디자인은 플레이를 더 잘 읽게 만들어야 한다.

### 4.3 세계관

작은 게임에도 한 문장짜리 세계관은 있어야 한다.

- 무엇을 하고 있는가
- 왜 바빠지는가
- 무엇이 좋아지는가
- 어떤 캐릭터나 사물이 기억나는가

세계관은 긴 스토리가 아니다. "라면집에 손님이 몰린다", "동물을 떨어뜨려 더 큰 동물로 합친다"처럼 플레이 행동을 더 맛있게 만드는 맥락이다.

### 4.4 사운드

사운드는 선택이 아니라 품질 요소다. 단, 항상 음소거 가능해야 한다.

- 입력음
- 성공음
- 콤보음
- 실패음
- 신기록음

사운드는 길면 안 된다. 반복 플레이에서 거슬리지 않아야 하고, 화면 반응과 박자가 맞아야 한다.

## 5. 게임 추가 기준

새 게임은 아래 질문에 답할 수 있어야 한다.

- 한 손으로 할 수 있는가?
- 첫 5초 안에 할 일을 알 수 있는가?
- 첫 30초 안에 재미가 오는가?
- 시간제한 없이 오래 생각하고 다시 돌아와도 이어서 할 수 있는가?
- 실패 이유가 명확한가?
- 더 잘하고 싶다는 여지가 있는가?
- 최고 기록으로 남길 만한 숫자가 있는가?
- 결과 화면에서 공유하고 싶은 순간이 있는가?
- 기존 포털 UI와 자연스럽게 붙는가?
- game kit 아이콘/오브젝트 체계로 표현 가능한가?
- 사운드 없이도 이해되고, 사운드가 있으면 더 좋아지는가?

이 질문 중 핵심 3개 이상이 애매하면 바로 구현하지 말고 프로토타입으로 검증한다.

## 6. 구현 체크리스트

새 게임을 추가할 때 확인한다.

- `games/{id}/` 폴더가 있다.
- `games.json`에 `id`, `title`, `tagline`, `unit`, `score_key`, `score_metric`, `state_keys`가 등록되어 있다.
- localStorage 키는 게임 id 접두사를 쓴다.
- 최고 기록은 `portal.js`가 잡을 수 있는 단일 키로 저장된다.
- 공통 셸에서 헤더, 푸터, 메뉴가 깨지지 않는다.
- 결과 화면에 다시 하기, 공유, 피드백, 후원이 있다.
- 기록실에서 게임명과 기록 단위가 자연스럽게 보인다.
- 모바일 세로 화면에서 주요 버튼이 엄지 영역에 있다.
- 시간제한, 방치 패널티, 강제 카운트다운이 없다.
- 페이지를 잠깐 벗어나거나 화면이 꺼져도 이어서 할 수 있다.
- 음소거 상태가 저장된다.
- 트레일·파편·셰이크·콤보 팝 등 주스(juice) 요소가 3가지 이상 들어 있고 판독성을 해치지 않는다.
- 아이콘은 포털 규칙을 따른다. 이모지를 UI 핵심 요소로 쓰지 않는다.
- game-icons.net 자산을 쓰면 푸터 크레딧을 표시한다.
- 플레이 중 DB/API 실패가 게임 진행을 막지 않는다.

## 7. 금지 사항

- 게임마다 다른 로그인, 기록, 후원, 공유 UI 만들기
- 이모지를 핵심 게임 오브젝트나 버튼 아이콘으로 쓰기
- 긴 설명을 읽어야만 시작할 수 있는 게임
- 양손 멀티터치를 요구하는 조작
- 작은 모바일 버튼에 핵심 기능 넣기
- 제한 시간, 방치 패널티, 강제 카운트다운으로 실패시키기
- 실패 이유를 숨기는 랜덤 이벤트
- 신기록 순간이 밋밋한 게임
- 사운드가 기본으로 너무 크거나 끌 수 없는 게임
- 포털 폰트와 다른 외부 폰트 사용
- 계좌, 토큰, 외부 키를 저장소에 커밋하기

## 8. 좋은 게임의 한 줄 정의

좋은 게임은 사용자가 설명을 읽기 전에 손이 먼저 움직이고, 실패한 뒤에는 이유를 알고, 끝난 뒤에는 바로 한 판 더 누르게 만든다.
